# CHANGE SPECIFICATION

> **PURPOSE** — Replace the heuristic keyword-matching engine in the "Generate Agent" feature with a real LLM call that receives full platform context (MCP catalog, agent families, skills, similar agents) and produces contextual, intelligent agent drafts. Retain the heuristic logic as a graceful fallback when the LLM is unavailable.

---

## 1. SUMMARY

The `POST /api/agents/generate-draft` endpoint currently returns `source="mock_llm"` but makes no actual LLM call. Agent drafts are generated by naive keyword-overlap scoring and fixed templates, producing shallow, context-unaware output. This change upgrades the generation pipeline to use a real LLM call that is aware of the full Orkestra ecosystem (MCP catalog, families, skills, similar agents) and produces high-quality, platform-coherent drafts. The frontend intent form is also enriched with explicit family and skill selectors so users can guide the LLM more precisely.

---

## 2. CONTEXT

### 2.1 Current State Snapshot

- `POST /api/agents/generate-draft` is handled by `agent_generation_service.generate_agent_draft()`.
- The service is fully synchronous and uses deterministic heuristics: keyword overlap to rank MCPs, hardcoded family inference rules, and a fixed generic prompt template.
- It always returns `source="mock_llm"` — a misleading label indicating intent but no real capability.
- The `AgentGenerationRequest` schema accepts `preferred_family` and constraints but ignores existing families, skills, and similar agents in the registry.
- The `orchestrator_builder_service` already implements the full async LLM call pattern (prompt building, `get_chat_model()`, `_parse_llm_json`, debug trace files) and serves as the established reference pattern.
- The frontend modal (`generate-agent-modal.tsx`) collects intent as free text only; no family or skill selectors are present.

### 2.2 Pain Points / Gaps

- Generated `prompt_content` and `skills_content` are shallow fill-in-the-blank templates disconnected from real platform vocabulary.
- MCP selection by keyword overlap is brittle; the LLM can reason semantically about MCP purposes.
- No awareness of existing agent families or skills means generated agents may reference nonexistent family IDs or duplicate skill definitions.
- Similar agents in the registry are ignored, leading to redundant or inconsistent drafts.
- `source="mock_llm"` is misleading and creates false expectations.
- The frontend form lacks controls for family and skills, forcing the LLM (once real) to guess without user guidance.

---

## 3. PROBLEM STATEMENT

The "Generate Agent" feature claims LLM-powered generation but delivers heuristic templates. The gap between the promise and the reality produces low-quality agent drafts that require significant manual editing, undermines user trust in the feature, and misrepresents the platform's AI capabilities. Bridging this gap requires both a backend service rewrite to perform a real LLM call with rich context injection, and a frontend improvement so users can meaningfully steer generation.

---

## 4. GOALS

1. Replace the heuristic engine with a real LLM call that uses platform context as input.
2. Return `source="llm"` for successful LLM-generated drafts.
3. Retain the heuristic path as a fallback that returns `source="heuristic_template"` when the LLM is unavailable.
4. Enrich the intent form with family selector and skills multi-select.
5. Surface MCP rationale from the LLM to the user in the review step.
6. Write debug trace files for LLM calls (same convention as orchestrator builder).
7. Keep all existing tests passing; add tests for the LLM path (mocked) and the fallback path.

### 4.1 Success Metrics / KPIs

| Metric | Target |
|--------|--------|
| `source` field value for LLM-backed calls | `"llm"` |
| `source` field value for fallback calls | `"heuristic_template"` |
| LLM path test coverage (mocked) | ≥ 1 test scenario covering happy path |
| Fallback path test coverage | ≥ 1 test scenario |
| `mcp_rationale` non-empty on LLM response | 100 % of LLM-sourced drafts |
| Existing generate-draft test suite | All passing (no regression) |
| P95 response time (LLM path, local model) | ≤ 30 s (agent creation is not a hot path) |

### 4.2 Non-Goals

- Streaming / SSE response for generation — out of scope.
- Automatic saving of the generated draft without user confirmation.
- Fine-tuning or hosting a custom model.
- Changing the `GeneratedAgentDraft` schema shape (additions are acceptable).
- Adding LLM generation to orchestrator builder (already done).

## 5. FUNCTIONAL CAPABILITIES

| ID | Capability | Rationale |
|----|-----------|-----------|
| F-1 | Real LLM call for agent draft generation | Core goal; replaces misleading mock path |
| F-2 | Full platform context injection into LLM prompt | MCPs, families, skills, and similar agents must be included so the LLM can produce coherent, non-redundant drafts |
| F-3 | Structured JSON response from LLM, validated against `GeneratedAgentDraft` | Ensures downstream consumers receive well-formed data regardless of LLM verbosity |
| F-4 | Heuristic fallback when LLM is unavailable | Maintains feature availability; `source="heuristic_template"` signals degraded mode |
| F-5 | `source` indicator surfaced to frontend | Users and operators can distinguish LLM vs. heuristic output |
| F-6 | Debug trace files for LLM generation calls | Operator observability; same format as orchestrator builder traces |
| F-7 | Frontend family selector in intent form | Allows user to constrain family, improving LLM output accuracy |
| F-8 | Frontend skills multi-select in intent form | Allows user to suggest skill assignments, improving LLM output accuracy |
| F-9 | MCP rationale displayed in review step | User can understand why each MCP was selected and toggle checkboxes with context |
| F-10 | Updated UI copy reflecting AI-powered generation | Truthful labelling now that actual LLM is used |

### 5.1 Capability Details

**F-1 — Real LLM call**
The generation service must become async and call the LLM using `get_chat_model()` with DB-sourced configuration (provider, model). It must follow the same invocation pattern as `orchestrator_builder_service` (prompt construction → LLM call → JSON extraction → schema validation).

**F-2 — Platform context injection**
Before calling the LLM, the API handler or service must fetch: (a) the full active MCP catalog, (b) all agent families, (c) all skills, and (d) up to 5 agents with semantic similarity to the intent (similarity heuristic or keyword proximity). These are serialised into the system prompt.

**F-3 — Validated JSON response**
The LLM is instructed to return a JSON object conforming to `GeneratedAgentDraft`. A `_parse_llm_json` helper extracts and validates the response; on parse failure the service falls back to the heuristic path (F-4).

**F-4 — Heuristic fallback**
The existing synchronous logic is preserved as a standalone function. It is called when: (i) no LLM is configured, (ii) the LLM call raises an exception, or (iii) JSON parsing fails. The response `source` field is set to `"heuristic_template"`.

**F-6 — Debug traces**
Trace files are written to `ORKESTRA_DEBUG_AGENT_GENERATION_DIR` (env var, default `/app/storage/debug-agent-generation`). Each trace file contains: request payload, context injected (MCP list, families, skills, similar agents), raw LLM response, parsed draft, timestamp.

**F-7 / F-8 — Frontend form enrichment**
The intent form gains: (a) a dropdown populated from `GET /api/families` for preferred family, and (b) a multi-select populated from `GET /api/skills` for preferred skills. Both are optional; selected values are sent in `AgentGenerationRequest.preferred_family` and a new `preferred_skill_ids` field (see DM-1).

**F-9 — MCP rationale in review step**
Each MCP checkbox in the review step is accompanied by the rationale string from `mcp_rationale[mcp_id]`. This is already present in the schema; the frontend must render it.

## 6. USER & SYSTEM FLOWS

### 6.1 Happy Path — LLM Available

```
User fills intent form
  → selects family (optional)
  → selects skills (optional)
  → clicks "Generate"

Frontend POST /api/agents/generate-draft
  ↓
API handler (async):
  1. Fetch MCP catalog from DB
  2. Fetch families from DB
  3. Fetch skills from DB
  4. Fetch similar agents (up to 5) from DB
  5. Call agent_generation_service.generate_agent_draft_llm(request, context)
     a. Read LLM config from DB
     b. Build structured system prompt with platform context
     c. Call LLM → raw JSON string
     d. Parse & validate → GeneratedAgentDraft
     e. Write debug trace file
     f. Return draft, source="llm"

Frontend receives AgentGenerationResponse (source="llm")
  → renders review step:
      editable fields (name, purpose, description, prompt_content, skills_content)
      MCP checkboxes with rationale tooltips
      source badge "AI-Powered"
  → user confirms / edits → saves draft
```

### 6.2 Fallback Path — LLM Unavailable

```
... (same steps 1–4 above)
  5. LLM unavailable OR call fails OR JSON parse fails
     → agent_generation_service._heuristic_generate(request, mcp_catalog)
     → source="heuristic_template"

Frontend receives AgentGenerationResponse (source="heuristic_template")
  → renders review step with source badge "Template (AI unavailable)"
```

## 9. NON-FUNCTIONAL REQUIREMENTS (NFRs)

| ID | Category | Requirement |
|----|----------|-------------|
| NFR-1 | Performance | P95 end-to-end latency for the LLM path ≤ 30 s on a locally-hosted model (agent creation is a low-frequency, human-triggered action) |
| NFR-2 | Performance | Heuristic fallback path must complete in ≤ 500 ms (unchanged from current baseline) |
| NFR-3 | Reliability | LLM call failures (timeout, connection error, invalid JSON) must never surface a 5xx to the client; fallback path must be invoked transparently |
| NFR-4 | Observability | A debug trace file must be written for every LLM call attempt (success or failure) within ≤ 1 s of response |
| NFR-5 | Maintainability | LLM prompt construction must be isolated in a single, testable function with no DB or I/O dependencies |
| NFR-6 | Testability | The LLM call must be injectable / mockable; no direct instantiation of the LLM client inside the service function under test |
| NFR-7 | Security | LLM prompt must not include raw user-controlled strings without sanitisation (e.g., strip control characters from `intent`, `constraints`) |

## 11. RISKS & MITIGATIONS

| ID | Risk | Impact | Probability | Mitigation | Residual Risk |
|----|------|--------|-------------|-----------|---------------|
| RSK-1 | LLM returns invalid or unparseable JSON | H — draft unavailable | M — LLM models vary in JSON reliability | Use `_parse_llm_json` with regex extraction; instruct LLM to return JSON only; fallback to heuristic on parse failure | L — fallback always available |
| RSK-2 | LLM response latency unacceptable for UX | M — user abandons generation | M — depends on model and hardware | Feature is human-triggered and non-blocking; add a loading indicator; 30 s timeout on LLM call | L |
| RSK-3 | LLM generates family_id or skill_id not present in DB | M — draft fails validation on save | M | Inject exhaustive family and skill lists in prompt; validate LLM output against DB lists; correct or nullify unknown IDs before returning | L |
| RSK-4 | Breaking change to `source` field breaks existing tests | L — test failures in CI | H — current tests assert `source="mock_llm"` | Update tests as part of this change | L |
| RSK-5 | Context payload too large for model context window | M — LLM truncates prompt | L — catalog typically small | Cap MCP catalog summary to essential fields; limit similar-agent block to 5 agents × 4 fields | L |

## 12. ASSUMPTIONS

1. `get_chat_model()` and `_read_llm_config_from_db` from the orchestrator builder pattern are available and reusable without modification.
2. `GET /api/families` and `GET /api/skills` endpoints exist and return stable list responses.
3. The LLM configured in the platform is capable of producing valid JSON when instructed.
4. Agent creation is a low-frequency, human-in-the-loop action; a P95 latency of 30 s is acceptable.
5. The MCP catalog is small enough (< 100 entries) that full injection into the prompt is feasible.
6. No migration of existing persisted data is required; `AgentGenerationRequest` is not stored.

## 20. PRIVACY / COMPLIANCE REVIEW

- User-provided `intent` text is sent to the configured LLM provider. If the provider is remote (non-Ollama), potentially sensitive business intent may leave the platform perimeter.
- Operators using a remote LLM should review their data-processing agreements before enabling this feature.
- Debug trace files contain the full request payload including `intent` and `constraints`; trace directory should be access-controlled.

---

## 21. SECURITY REVIEW HIGHLIGHTS

- `intent`, `constraints`, and other free-text request fields must be sanitised (strip control characters, enforce max-length) before inclusion in the LLM prompt to prevent prompt injection.
- LLM API keys / credentials must be read from DB config, never hardcoded or logged.
- Debug trace files must not be served over HTTP; they are operator-only filesystem artefacts.

---

## 22. MAINTENANCE & OPERATIONS IMPACT

- New environment variable `ORKESTRA_DEBUG_AGENT_GENERATION_DIR` (optional, documented).
- Trace files accumulate on disk; operators should configure log rotation or periodic cleanup for the trace directory.
- LLM provider configuration (provider, model, API key) is shared with the orchestrator builder; no new DB config keys required if the provider is already configured.

## 24. APPENDICES

### Appendix A — Current Heuristic Scoring Logic (for reference)

The existing `_rank_mcps` function assigns scores based on: keyword overlap between intent and MCP metadata (×4), Orkestra-enabled state (+3), Obot-active state (+2), preferred MCP scope match (+3), and effect-type bonuses for search/lookup (+1). This logic is preserved as the fallback path under `source="heuristic_template"`.

### Appendix B — `orchestrator_builder_service` Pattern Reference

Key patterns to replicate:
- `_read_llm_config_from_db(session)` → returns `(provider, model, api_key)`
- `get_chat_model(provider, model, api_key)` → returns LangChain-compatible chat model
- `_parse_llm_json(raw_text)` → extracts JSON from LLM response, returns dict or raises
- `_write_trace(filename, data)` → writes JSON to configurable trace directory
